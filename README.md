# BEET 2.0 — LLM Authorship Detection

A multi-detector fusion pipeline for classifying whether a piece of text
was written by a human or generated by a large language model. BEET
combines pattern-based, statistical, and LLM-backed detectors into a
single calibrated verdict with a confidence interval and per-feature
contributions.

The project ships three runtimes that share one pipeline:

- **`beet gui`** — local web UI with tabs for analyze, batch, history,
  monitoring, evaluation, and settings.
- **`beet serve`** — FastAPI REST service for integrations.
- **Tauri desktop shell** — packaged `.msi` / `.dmg` / `.AppImage`
  wrapping the same UI over a stdio sidecar.

## Quick start

```bash
pip install -e .

# web UI on http://127.0.0.1:8877
beet gui --profile screening
```

Paste text, click Analyze. The result panel shows:

- **Determination** — RED / AMBER / YELLOW / GREEN / UNCERTAIN / MIXED
- **P(LLM)** with a conformal confidence interval
- **Top contributing signals** from the EBM fusion model
- **Detector breakdown** — per-layer raw score, calibrated p(LLM),
  confidence, verdict, and signals
- **Highlighted text** — character-span overlays showing which phrases
  triggered which detector

## What's in the box

### Detectors

Sixteen detectors organised into four cascade phases:

| Phase | Cost | Detectors |
|------:|------|-----------|
| 1 | trivial / cheap | `preamble`, `fingerprint_vocab`, `prompt_structure`, `voice_spec`, `instruction_density`, `nssi` |
| 2 | moderate | `surprisal_dynamics`, `contrastive_lm`, `token_cohesiveness` |
| 3 | expensive (LLM-backed) | `perturbation`, `contrastive_gen`, `dna_gpt` |
| 4 | batch-only | `cross_similarity`, `mixed_boundary`, `contributor_graph` |

The cascade scheduler short-circuits on confident Phase 1 verdicts and
only invokes expensive Phase 3 detectors when the cheaper signals are
inconclusive.

Six of the detectors emit **character-level spans** pointing at the
specific tokens that triggered them (preamble, fingerprint_vocab,
instruction_density, nssi, voice_spec, prompt_structure) — surfaced as
highlights in the UI.

### Fusion, calibration, decision

- **EBM fusion** (interpret-ml Explainable Boosting Machine) combines
  per-detector `p_llm` scores with calibrated contributions.
- **Isotonic calibration** per detector from a labeled dataset.
- **Conformal prediction** for a valid P(LLM) confidence interval.
- **Decision engine** maps fused P(LLM) to RED / AMBER / YELLOW / GREEN
  with configurable thresholds, abstains to UNCERTAIN when the
  prediction set is too wide, and enforces override rules (e.g. a
  CRITICAL `preamble` match short-circuits to RED).

### Built-in observability

- **SQLite history store** — every analyze/batch call is logged and
  queryable by the History tab.
- **Monitoring dashboard** — determination distribution, per-day
  volume, 200-point recent-submissions scatter, per-detector
  aggregate stats, feedback-derived accuracy and Brier score.
- **Drift monitor** — KL divergence per feature against a baseline you
  seed from recent history; population and calibration drift alerts.
- **Evaluation runner** — wraps `beet.evaluation.runner.run_eval`
  inside the UI with AUROC / ECE / Brier / TPR@FPR / per-tier and
  per-attack breakdowns plus a confusion matrix.
- **Hash-chained audit log** for provenance (`beet.provenance.chain`).

## Repository layout

```
beet/                        Python package
  pipeline.py                 Orchestrates router → cascade → fusion → decision
  router.py                   Classifies text as prompt / prose / mixed
  cascade.py                  Phase scheduler
  fusion/                     EBM model + conformal prediction
  decision.py                 Threshold + abstention + override engine
  detectors/                  16 detector modules
  evaluation/                 Dataset loader, runner, metrics, ablation
  monitoring/                 Drift + meta-detector health
  privacy/                    Feature store, retention, access log
  provenance/                 Hash-chained audit log
  history.py                  SQLite submission log (queryable index)
  runtime.py                  Pipeline/profile/config hot-swap context
  sidecar.py                  stdio JSON-RPC for the Tauri shell
  api.py                      FastAPI app
  gui/server.py               stdlib HTTP server
  gui/static/index.html       single-file SPA (used by all three shells)
  console.py                  beet CLI entry point
configs/                     YAML profiles (default, screening, strict, no-api, production)
src-tauri/                   Tauri 2 desktop shell (Rust)
docs/                        Documentation (see below)
scripts/                     Training: calibrate_detectors.py, train_fusion.py, ...
tests/                       ~310 unit + integration tests
```

```bash
pip install -e ".[dev]"
pytest
```

Around 310 tests covering contracts, detectors, router, cascade,
fusion, decision, calibration, evaluation, monitoring, privacy,
provenance, history, the HTTP API, and the sidecar RPC.

## Development loop

For the desktop shell, see `docs/building.md`. For web-UI changes,
`beet gui` reloads `index.html` on every request (it's a static file),
so frontend iteration is a browser refresh.

For backend changes, restart `beet gui`. The stdio sidecar is
restarted by the Tauri shell on relaunch.

## Limitations

- Tier-3 detectors (`contrastive_gen`, `dna_gpt`) require API
  credentials (`pip install -e ".[tier3]"` plus env vars).
- `run_eval` inside the GUI is capped at 1000 samples; larger runs
  should drop to the Python module (`beet.evaluation.runner.run_eval`).
- DriftMonitor observations are in-memory — the baseline is rebuilt
  from the persistent history store on demand, but the current window
  is lost on sidecar restart.
- Trained fusion / calibration artifacts are referenced by config but
  not checked into the repo. See `scripts/` for training recipes.
