# CLAUDE.md — BEET 2.0 guidance for Claude Code

This file tells Claude Code how to work in this repo. Follow it as though
these were your standing orders here.

## What this project is

A multi-detector fusion pipeline for classifying whether text was written
by a human or generated by an LLM. Three runtimes share one SPA:

- `beet gui` — stdlib HTTP + `beet/gui/static/index.html` (single file SPA)
- `beet serve` — FastAPI (needs `beet[api]`)
- Tauri desktop shell — `src-tauri/` wraps a `beet.sidecar` subprocess over stdio JSON-RPC

See `README.md` for the user-facing description, `docs/architecture.md`
for internals, `docs/configuration.md` for profile tuning.

## Invariants — do not break

### Honesty about calibration

The system is in heuristic mode by default — no trained EBM fusion, no
calibrated conformal wrapper, hand-picked per-detector p(LLM) tables. The
UI announces this with a banner; `/health` reports
`calibration_status: "heuristic"`.

- **Detector-internal constants that map raw signal → p(LLM) must be
  named `_HEURISTIC_*`** (see `fingerprint_vocab.py`, `instruction_density.py`,
  `preamble.py`, `perturbation.py`, `contrastive_gen.py`). Do not relabel
  them as "calibration" tables — that reads as empirical ground truth
  when they are not.
- **`prediction_set` is cosmetic severity bands** (RED/AMBER/...).
  **`conformal_set` is the honest binary set** (`["LLM"]` / `["human"]` /
  `["human", "LLM"]`) — populated only when a calibrated conformal wrapper
  is loaded. Never conflate them.
- **`fusion_mode: "naive"` → "contributions" are deviations-from-prior**,
  not additive feature attributions. The UI caveats this; preserve the
  distinction in any new surface.

### Detector failures surface, not swallow

`pipeline._run_phase` logs and records detector failures to
`Determination.detector_errors` (+ UI banner). Do not add bare
`except Exception: pass` around detector calls. If a new detector can
fail partially, include enough in the `signals` dict or `error` field
to diagnose.

### Privacy boundaries

- `contrastive_gen` sends **only the task description** to the provider,
  never the submission text. There's a test asserting this — keep it
  passing.
- `gui.history.retain_text` defaults to **false** (hash-only). Don't
  flip the default without flagging it as a privacy change.
- `_redact` in `beet/api.py` walks the full config and scrubs keys
  matching: `secret, api_key, apikey, token, password, passwd,
  credential, auth`. Extend if you find new sensitive key patterns.

### Cascade-phase membership

`beet/cascade.py::PHASE{1,2,3,4}_DETECTORS` is the source of truth for
what runs when. If you change it, update `README.md`'s Phase table
accordingly (they drifted apart once and got a review callout).

## Code conventions

- **Python 3.11+ with type hints on public surfaces.** Dataclasses for
  contracts (`beet/contracts.py`); `Protocol` for structural typing.
- **Stdlib at boundaries.** `http.server`, `sqlite3`, `json`. External
  deps are fine for algorithms (`interpret`, `scikit-learn`,
  `transformers`, `sentence-transformers`) behind extras.
- **Optional deps load lazily.** `torch`, `transformers`, `anthropic`,
  `openai`, `sentence-transformers` — import inside the function that
  needs them. The base install must stay tiny.
- **Dataclasses add fields with defaults**, never reorder existing
  ones. `Determination` and `FusionResult` are consumed by tests,
  reports, and the UI.
- **Best-effort writes** for history, drift, audit. A failed side-channel
  write must never fail the analyze RPC. Log to stderr and continue.

## Tests

`pytest` with ~35 test files covering contracts, detectors, router,
cascade, fusion, decision, calibration, evaluation, monitoring,
privacy, provenance, history, HTTP API, sidecar RPC, availability
introspection, evaluation serialization.

Structural tests (did the field land, did the RPC round-trip, does
this detector fire on this fixture) — **they do not establish
empirical accuracy**. Don't write tests that tautologically assert
the heuristic constants (e.g. `p_llm >= 0.75` because the preamble
table says so). Write tests that assert *behavior* — spans land,
errors surface, conformal set is binary when calibrated.

Run the full suite: `pytest -q`. Expect ~340 passing.

## Adding a detector

1. Create `beet/detectors/your_detector.py` implementing the
   `Detector` protocol from `beet/contracts.py`.
2. Register in cascade (`beet/cascade.py::PHASE{1,2,3,4}_DETECTORS`).
3. Add default entry in `configs/default.yaml::detectors`.
4. Add a test in `tests/test_detectors/test_your_detector.py`.
5. If the detector has positional evidence (regex matches, token
   offsets), populate `spans` on the `LayerResult` and add a
   `.span-<kind>` CSS class in `beet/gui/static/index.html` if your
   `kind` isn't already in the palette.
6. If the detector needs extras, list them in `pyproject.toml` under
   the relevant extras group (`tier2`, `tier3`, etc.) and have the
   module degrade with a clear SKIP signal when the import fails.
7. Update `beet/pipeline.py::detector_availability` if the detector
   has a runtime requirement that can't be detected by registry-load
   alone (API key, batch context, etc.).

## Adding an RPC method — 4 touch points

Both transports route to shared code. A new method needs:

1. Method body on `Sidecar` class in `beet/sidecar.py` (or a helper).
2. Dispatch entry in `Sidecar.handle`.
3. HTTP route in `beet/gui/server.py::_make_handler`.
4. Tauri command in `src-tauri/src/commands.rs` + registration in
   `src-tauri/src/main.rs::invoke_handler`.

Plus tests in `tests/test_rpc_sidecar.py` and `tests/test_http_api.py`.
Plus `beetApi` entry in both branches of `beet/gui/static/index.html`.

## Design specs before non-trivial code

Non-trivial features get a spec under
`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` before the code.
See that directory for worked examples (six GUI phases, drift wiring,
etc.). Skip for one-line fixes; mandatory for anything that changes
contracts, endpoints, or user-facing surfaces.

## Working with the UI

`beet/gui/static/index.html` is one big file by design. When adding UI:

- API client entries go in **both** branches of `beetApi` (Tauri
  `invoke` and HTTP `fetch`) — keep the signatures identical.
- Tabs register in the `VIEWS` array + the `showTab` switch and get a
  `loadX` loader invoked when the tab becomes active.
- Span highlight colors use the `.span-<kind>` CSS pattern; new kinds
  add a class + a chip in the legend's `KIND_LABELS` map.
- Don't break the `window.__TAURI_INTERNALS__` detection — the single
  file has to work in both fetch-mode (web) and invoke-mode (Tauri).

## Things the project does NOT have yet (don't pretend it does)

- A trained EBM fusion model — `fusion.model_path` is unset in every
  config, and no artifact is checked in.
- A calibrated conformal wrapper — `fusion.conformal_path` same story.
- A labeled dataset of any size in the repo. `scripts/download_hc3.py`
  can fetch one; run it before doing any calibration work.
- A production CI smoke number beyond HC3 finance screening
  (`docs/benchmarks/hc3_finance_screening_smoke.json`). Other
  profiles / datasets are future work.
- Auth / rate limiting on the FastAPI or GUI HTTP layers. Binds to
  loopback by default; hardening is on the roadmap.
